## 1. Table Decisions

| Question | Decision |
|---|---|
| What proves promotion? | Body-grown lives under explicit tree authority |
| May synthetic fixtures count? | Only for unit shape; never for route, death, grave, or lineage promotion |
| Required deterministic stratum | L0, permanent and reproducible |
| Required live stratum | L1, bounded manual DeepSeek evidence before Step 5 |
| Edge threshold for full-tree default | All 38 legal directions across all 22 canonical edges executed under tree authority |
| May topology imply an unexecuted direction? | No |
| May committed-only count as coverage? | No; it is a separate host/failure boundary witness |
| Is legacy agreement success? | No; legacy is comparison evidence, not oracle |
| Observer ablation | Required for every deterministic corpus family |
| May a tie of constant witnesses count as task-sensitive routing? | No |
| Default change in this step | Forbidden |

The 38-direction requirement is intentionally strict. A default called
`full-tree` cannot silently promote only the currently convenient fragment of
the Tree. If a legal direction cannot be grown, Step 5 remains blocked until
the topology or declared authority surface is explicitly revised in a separate
design process.

## 4. Per-Life Evidence Record

Every L0/L1 life produces one bounded record with these fields:

| Group | Required fields |
|---|---|
| Identity | case id, Packet id, lineage id, generation, session id |
| Invocation | task class, prompt hash, work mode, substrate/model, router mode, observer mode |
| Economics | initial/remaining/spent budget, token truth status, loss limit/current/events |
| Motion | walk, tick count, current/final operator, stop reason |
| Derivation | pressure snapshot ref, all contributions, candidate audit, selected readiness, derivation ref |
| Ledger | candidate, selected, committed, executed, failed counts and refs per direction |
| Observer | agreement/dissent/unavailable, predicted target/reason, observed authority |
| State | field/relation/revision deltas, progress, validation/effect evidence |
| Finality | manifest status, terminal kind/cause, death, residue, grave |
| Lineage | inherited records, pressure contributions, generation delta, control-line delta |
| Reader audit | writer, record kind, named reader, activation law, consumed ref/status |
| Artifact | deterministic test path or dated live log path |

Absence is represented explicitly as `none`, `not_activated`, `unsupported`, or
`missing_reader`. Empty cells are forbidden.

## 5. Ledger State Table

| State | Meaning | Counts as executed edge coverage? |
|---|---|---|
| candidate | Neighbor evaluated by tree policy | No |
| selected | Tree chose candidate before commit | No |
| committed | Packet position changed and route event exists | No |
| executed | Destination organ completed its tick | Yes |
| failed | Destination produced typed effect failure | No; separate failure evidence |
| pending_at_host_ceiling | Route committed, destination not ticked | No; required P08 evidence |
| excluded | Candidate failed availability/readiness/affordability/topology | No |

An execution reference must point to the destination tick/effect. A route event
alone cannot create executed credit.

### A1.1 Phase order

The promotion campaign now has two distinct evidence phases:

```text
W0 witness repair and matched controls
-> C0 deterministic promotion corpus
-> C1 live substrate corpus
-> promotion record
```

W0 and C0 may use the same body APIs, but they cannot be the same fitting
dataset.

```text
W0 asks: does this sensor track its named fact?
C0 asks: can the complete body grow the required lives and directions?
```

The corpus must not choose witness definitions, coverage domains, weights, or
thresholds from the same route outcomes later presented as promotion evidence.

the topology or declared authority surface requires a separate
chaos -> table -> crystall revision. There is still no `accepted_exception`.
