## 1. Question

Roadmap step 4 gave the body exact object/version coverage and one complete
opt-in vertical Packet life. It deliberately did not answer three scheduling
questions:

```text
when does a relation coverage gap become a real need for CONNECT?
which exact field mutation creates a real need for upper OBSERVE?
how do several valid needs compose without vibed weights or canonical theater?
```

These are not one question about coefficients. They are three boundaries:

```text
fact -> need
need -> executable action
several actions -> route or honest ambiguity
```

## 4. Relation Need Is A Join, Not A Gap

Selected working hypothesis:

```text
relation_need = exact coverage delta
              AND exact body-visible candidate/demand
              AND named live consumer
```

The three terms answer different questions:

| Term | Question | Current owner |
|---|---|---|
| coverage delta | Has current CONNECT policy inspected these object versions? | object coverage |
| candidate/demand | Is there a specific relation-shaped fact or request over those objects? | pure CONNECT inspection / future typed demand |
| consumer | What existing operation changes or blocks because of that relation? | relation-guided ENCODE v0 |

Coverage alone remains CONNECT readiness:

```text
CONNECT can inspect this changed scope
```

It does not become pressure:

```text
CONNECT should inspect this scope now
```

### 4.1 Causal affordance v0

The first accepted candidate class is a deterministic structural relation that
the body can preview without mutation:

```text
explicit parent/carrier structure
registered L1 projection relation
another registered body-native predicate with exact provenance
```

If that candidate is absent from the current exact raw epoch and
relation-guided ENCODE can consume its kind, it may create a
`relation_affordance_need` toward ☰.

The witness must name:

```text
candidate kind
endpoint ids and current versions
candidate predicate/source events
consumer contract id
coverage delta refs
```

A new unit that creates no new candidate produces no relation pressure. This
directly rejects both the single-sample empty probe and the post-ENCODE formed
unit recurrence observed above.

### 4.2 Blocking demand remains possible but absent by default

A future body-owned record may state that a particular operation cannot
proceed without a relation of a declared kind. Such a witness is stronger than
an affordance, but it must pass the existing starvation acceptance law.

Neither generic missing connections nor prose saying "relations would help"
is a blocking demand. A semantic substrate proposal may propose a demand, but
its applicability remains estimated/semantic until the body validates the
referents and consumer contract.

### 4.3 The need transforms after CONNECT

When ☰ records the candidate, the pre-CONNECT witness is discharged. The same
causal chain may then derive a new action:

```text
current supported raw relation + no terminal disposition
-> relation_formation_need toward ☵
-> exact relation_input action plan
-> relation-guided ENCODE
-> raw phase encoded; need discharged
```

This is not one persistent mutable need object. Each phase is freshly derived
from field plus immutable trace.

## 5. Upper Sight Is Delta Plus Sensor Plan

The upper-eye defect is now narrower than the old "OBSERVE always becomes
stale" story. Object-version coverage already proves the required spatial
fact. What is missing is a pressure reader and an executable body-owned sensor
plan over the same refs.

Selected working hypothesis:

```text
upper_observation_need = significant exact object/version delta
                       AND a sensor able to read that delta
```

Initial sensor classes:

| Delta class | Sensor | Substrate |
|---|---|---:|
| unresolved semantic ingress | `semantic` | yes |
| formed/selected/suppressed/dissolved field consequence | `field_native` | no |
| exact raw relation requested for inspection | `relation_native` | no |
| unknown/unclassified mutation | no promoted pressure; typed diagnostic | no |

The body chooses the sensor from Packet facts. The LLM fills only the semantic
sensor. It does not choose that OBSERVE is the next operator.

### 5.1 Coverage must be composite and derived

Different sensor modes inspect different scopes. Upper freshness therefore
cannot mean "take only the latest observation record". It needs a bounded
derived view over immutable upper observations:

```text
for each current field object
  find the newest observed exact version under a compatible sensor policy
compare that derived coverage with current object/version state
```

No mutable `needs_observation` flag or second coverage ledger is introduced.

### 5.2 Own output is covered atomically

Semantic OBSERVE creates one proposal unit. The observation and planned unit
must form one body-owned commit for coverage purposes:

```text
the input scope is observed
the planned sensor output is recorded as covered at version 1
the output cannot immediately request another identical OBSERVE
```

Expected validation failure mutates nothing. An impossible failure after the
first append is a loud invariant failure, not Packet death.

## 6. A Route Needs An Action Plan

The current tree router selects a glyph while detailed organ options still
come from the harness. That boundary was sufficient for shadow routing, but it
cannot execute the new witnesses autonomously.

A qualified contribution must therefore carry a body-derived bounded action
plan:

```lua
{
  witness_id = string,
  target_operator = glyph,
  causal_class = "blocking_demand" | "causal_affordance",
  scope_refs = string[],
  provenance_refs = string[],
  action = {
    mode = string,
    -- exact CONNECT policy/scope, OBSERVE sensor/scope,
    -- or ENCODE relation_input
  },
  discharge = {
    reader = string,
    expected_record = string,
  },
}
```

The action is not a tool call from the substrate. It is a deterministic body
projection of the same Packet facts that produced the witness.

Required closure:

```text
pressure scope refs == readiness scope refs == action scope refs == effect scope refs
all additional provenance refs resolve, but they are not fake organ operands
```

If one target has several incompatible plans, the body reports
`ambiguous_action`; it does not silently choose one by table order.

## 7. Composition Before Calibration

The Packet model calls `Tau` a per-edge pressure snapshot. It does not require
that every physical distinction be collapsed immediately into one scalar.

For step 5, pressure is better represented as a bounded set of qualified
obligations than as invented magnitudes:

```text
hard lifecycle/safety law removes impossible candidates
terminal manifestation is a separate highest boundary class
blocking demand strictly dominates an optional affordance on the same decision
independent equal-class demands without measured dominance remain ambiguous
canonical order may keep legacy/control moving but cannot qualify promotion
```

This is a partial order, not a claim that the final pressure model is solved.

Rejected shortcuts:

```text
count refs and call a larger set more urgent
assign 1/2/3 weights because they look reasonable
let canonical order masquerade as physical selection
introduce age before stable witness identity and discharge exist
force every legal edge merely to close 38/38
```

Age/fairness remains a later scheduling question. First the body must possess
varying witnesses worth scheduling.
