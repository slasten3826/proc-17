## State In One Sentence

proc-17 can live, structure, choose, observe, validate, die, leave residue and
continue one task through a clean descendant, but it cannot yet perform
repository work with body-owned hands.

It is therefore a working process-physics engine and the beginning of a coding
agent, not yet a complete coding agent.

## What Is Implemented And Measured

### One mortal Packet life

- Packet state owns CHAOS, CALM, BOUNDARY, trace, budget, identity loss and
  residue.
- Every body event carries an epistemic status. Substrate output remains a
  semantic proposal; only the body may confirm runtime facts.
- Budget exhaustion, identity loss, completion, typed effect failure and a
  coherent lack of viable movement can terminate a Packet.
- A terminal Packet is final. Its trace and corpse cannot be rewritten by later
  body operations.
- LOGIC distinguishes a rejected external effect from a broken Lua world. A
  typed world failure may become Packet mortality; invariant and harness faults
  remain loud and cannot enter graves as invented experience.

### Operator body and movement

- All ten ProcessLang operators are registered behind one body-owned dispatch
  boundary.
- The canonical tree contains all 22 transitions.
- The deterministic Tree router derives movement from Packet facts, pressure,
  readiness, capability, affordability and topology. The LLM does not select
  the next operator.
- Tree authority has executed a complete opt-in life. The default remains
  `shadow`: legacy movement stays in control while Tree predictions are
  measured without physical mass.
- The old mandatory eye rails remain historical scaffolding, not the final law
  of movement.

### Packet Slop L1/L2 joined to the CALM body

- Session-owned L1 marks every birth without becoming a mortal Packet organ.
- FLOW materializes one user or lineage ingress as new potential.
- CONNECT creates a bounded transient raw-relation epoch.
- OBSERVE, DISSOLVE and ENCODE implement the three dispositions of that raw
  epoch: inspect it, release it, or form it.
- Per-object versions and coverage prevent global revision noise from
  pretending to be a real witness.

### Exact structure, choice and plan delivery

- Under qualified pressure, ENCODE accepts one strict machine proposal and
  forms exact CALM work, field units, identity provenance and explicit loss.
- CHOOSE operates on one declared alternative set, selects one current member,
  records every suppressed alternative and advances exact object versions.
- Choice loss is visible and proportional to killed alternatives in the current
  provisional model.
- RUNTIME can assess one complete exact plan from Packet state.
- MANIFEST can project that assessment as terminal `plan.result.v0` without
  reading runner-local output or copying raw substrate prose as runtime truth.

### Eyes, memory and mortality

- OBSERVE provides upper semantic and field-native sight.
- RUNTIME is a continuous camera with immutable frames and explicit
  reconciliation, not an LLM-powered second eye.
- Graves classify deaths as warnings, bequests or neutral completion.
- Descendants can inherit warning pressure; old graves compost into bounded
  aggregate session patterns.
- Memory is session-scoped rather than globally shared.

### First task lineage across deaths

- One Packet is one mortal generation. One lineage is the task ancestry across
  terminal Packet lives.
- The outer lineage runner owns generation identity, parent/corpse/carrier
  references, a one-child-per-corpse law and cumulative economics.
- A terminal Packet becomes an immutable, hashed corpse.
- A deterministic bounded carrier crosses the boundary through `NETWORK@▽`.
  NETWORK is a boundary transformation, not an eleventh Packet operator.
- The descendant receives a new Packet identity and exact ancestry while
  starting without its parent's CALM, relations, operator state, local loss or
  runtime ledgers.
- A grown two-generation test proves the first complete lineage: generation 1
  dies from its own budget; generation 2 enters through the carrier, starts
  clean and completes the exact plan.

## What Is Not Implemented

### No body-owned repository hands

proc-17 can form and select work, but it cannot yet mutate a repository and
feed the resulting evidence back into its own Packet. Current coding batteries
manifest code through proc-17, while an external harness writes files and runs
checks. This proves useful delivery, not autonomous repository work.

### The existing sandbox is not a valid hand boundary

- Current path checks are lexical and need protection against symlink escape.
- Existing spell execution touches shell-oriented mechanisms.
- Old filesystem and spell helpers must not be connected directly merely to
  make a demonstration pass.

### Incomplete physical coverage

- Qualified pressure covers only selected relation, sight, structure, choice
  and plan-delivery chains.
- Generic semantic ENCODE, qualified DISSOLVE, build execution and most lower
  work still need exact witnesses and action contracts.
- Pressure coefficients are uncalibrated controls, not measured natural
  constants.
- The full promotion corpus is incomplete, so Tree authority must remain
  explicit rather than silently becoming the default.

### Incomplete ancestry and memory

- Lineage is in memory only: there is no crash recovery, automatic resume,
  descendant branching or provider-owned substrate conversation continuity.
- Exact lineage completion currently understands `plan.v0`, not `build.v0`.
- Bequests and compost patterns still lack general named readers beyond the
  bounded inheritance already exercised.

### No product surface

The machine CLI and human Go TUI remain designs. Building them now would expose
an engine that can explain intended work but still requires an external actor
to perform it.

## Next Product-Bearing Pressure

The next bounded layer is a capability-safe repository hand.

The first experiment should close exactly one real work loop:

```text
☵ forms one exact executable work unit
☳ selects that work unit
HAND mutates only the granted sandbox capability
☶ reads reality back and records fresh evidence
☱ confirms actual progress
the work unit transitions to done
☲ continues or stops from remaining work
△ assembles the verified result
```

The smallest honest v0 is not arbitrary shell access. It is one exact file
operation inside one explicit repository root, followed by independent
body-owned read-back evidence. The test corpus must also attempt absolute-path,
`..`, symlink and ungranted-command escapes and prove that all are rejected.

Expected environmental failure and broken body physics must remain separate:

```text
well-formed denied/failed effect -> typed effect_failure
Lua error / malformed effect / invariant violation -> loud harness failure
```

## Proposed Work Order

1. Audit `core/sandbox.lua`, `tools/fs.lua`, `logic/spells.lua` and every current
   path from selected work to external mutation.
2. Write a table contract for capability identity, repository root, granted
   operations, path resolution, effect records, evidence and failure classes.
3. Crystallize one exact action envelope and one independently verifiable
   result envelope.
4. Implement the hand behind an explicit opt-in policy. Do not grant it to the
   default router during construction.
5. Grow one repository task inside `sandbox/` and the corresponding escape and
   malformed-effect cases.
6. Manifest the treatment only after the body itself can move the work unit to
   `done` from fresh evidence.

Persistence, semantic carrier compaction, branching, CLI and TUI are not the
next pressure. They may improve continuity or usability, but none closes the
missing work loop.

## Review Request For Another Machine

Do not review this project from documentation alone. Read the implementation,
run the complete test corpus, and reproduce at least one claimed life. For this
handoff, the highest-value review questions are:

1. Can one capability object prevent lexical, canonicalization and symlink
   escape without creating a second authority over Packet state?
2. What is the smallest exact repository mutation that can produce independent
   LOGIC evidence and a body-owned `done` transition?
3. Which existing helper would violate that boundary if reused unchanged?
4. Is there a concrete defect in the current lineage or completion claims that
   can be reproduced from a grown life rather than a synthetic fixture?

Praise without a reproduced defect or explicit residual risk is not sufficient
review evidence.
