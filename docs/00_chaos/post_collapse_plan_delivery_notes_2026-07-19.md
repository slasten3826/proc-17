## Topology Correction

The first shorthand was:

```text
complete plan -> △
```

That shorthand is not a legal same-life route from the current position.
ProcessLang has no `☴-△` edge. The lawful path is:

```text
☴ -> ☱ -> △
```

The lower eye is not decorative. It is the boundary where current structured
potential is reconciled into a body-owned completion assessment before the
terminal assembler receives authority.

Therefore the treatment is one causal chain with two actions:

```text
exact plan candidate
  -> plan_completion_review need
  -> ☱
  -> runtime-confirmed plan completion assessment
  -> plan_delivery need
  -> △
  -> Packet-local manifest payload
  -> terminal death
```

No direct `☴ -> △` exception may be added to topology.

## Four Gaps Hidden Inside One Stall

### 1. Work mode is harness context

`options.work_mode=plan` reaches prompts and several organ calls, but the born
Packet does not own it. Both `metadata.work_mode` and `regime.work_mode` are
absent in the grown life.

A body-owned completion reader cannot ask the runner what kind of life it is.
Plan/build intent must become an immutable birth fact before it can cause
pressure.

### 2. Qualified pressure has no lower reconciliation word

The continuous camera already sees significant post-ENCODE and post-CHOOSE
effects, and ☱ readiness is true. `qualified_need_v0` nevertheless produces no
witness for ☱ because its vocabulary currently ends at relation, upper sight,
structure formation and alternative collapse.

The debt exists. Its qualified producer is missing.

### 3. Generic progress confuses plan content with build work

Exact plan items are stored as pending CALM work units. `body.progress` therefore
correctly says they have not been executed, but plan mode does not ask the body
to execute them. They are the requested outward artifact.

Changing those units to `done` would be a lie. Plan completion requires a
plan-specific assessment, not mutation of build progress semantics.

### 4. MANIFEST readiness and material are disconnected

Current MANIFEST readiness accepts almost any historical source ref. At the
same time, its text projection looks at the latest OBSERVE tick carried by
`options.result`. The final exact observation is field-native and contains no
substrate text, so the organ is broadly ready to assemble an empty plan.

This is the known harness-memory defect at the exact point where it becomes
observable. Production plan delivery must read bounded Packet records and the
committed action only.

## Central Law

```text
The substrate may propose plan content.
Only the body may confirm that the plan form is complete enough to deliver.
Only MANIFEST may assemble the outward artifact.
```

Forbidden completion evidence:

```text

MANIFEST readiness finds any old source ref
CALM work units are relabelled done without execution
caller passes result ticks or selected ids
```

Absence of known pressure is not completion. Completion is its own positive,
bounded witness.

## Work Mode As A Birth Fact

The first treatment needs exactly two accepted modes:

```text
plan
build
```

The value originates at the user/CLI/session boundary, but after birth it
belongs to the Packet. It is not an LLM classification and not a mutable
router option.

Provisional body location:

```text
packet.regime.work.mode
```

Compatibility may mirror it to `metadata.work_mode` while old readers exist.
For old direct callers, a valid metadata mode may seed the canonical field when
no canonical option is supplied. If caller metadata, packet options and runner
options disagree, birth fails loudly. The runner may transport the requested
mode into birth but cannot change it after birth.

No plan completion witness exists in build mode.

## Phase A: Plan Completion Candidate

A bounded v0 candidate is derived from Packet state only. It requires:

```text
Packet regime says plan
exact structure inspection is complete and untruncated
exactly one current structure formation is selected for delivery
formation has no repair status
formation identity map and linked loss proof resolve
all formed unit ids still exist at their exact current versions
latest upper coverage includes every current formed version
no current structure_formation need remains
no current choice_need remains
if shape is alternative_set, exactly one member is selected
all other members of that set are explicitly suppressed
post-collapse versions of selected and suppressed members were observed
latest applicable validation is not rejected
no unresolved typed operator failure targets the candidate
```

For the first implementation, multiple current exact formations are typed
`ambiguous_plan_material`. They do not merge implicitly.

An encoding loss record with omitted items or omitted edges makes the candidate
`partial`, not `complete`. A future partial-delivery policy may manifest it with
residue, but this treatment does not silently call it complete.

Accepted shapes for investigation:

```text

artifact_set

```

The table may narrow this set if one shape lacks a complete Packet-local proof.

## Phase B: ☱ Completion Review

From the final ☴ position, a complete candidate does not immediately become a
terminal fact. It creates one blocking `plan_completion_review` witness toward
☱.

The witness must name:

```text
Packet id and generation
body-owned work mode
structure formation event
exact formed unit ids and versions
latest covering upper observation refs
choice event and post-choice versions when applicable
linked crystallization, identity map and loss refs
significant pending runtime frame refs
relevant Packet revisions
```

The action must consume the same scope. ☱ performs its normal camera
reconciliation and writes one additional immutable assessment:

```text
protocol_version = plan.completion_assessment.v0
state = complete | partial | blocked | ambiguous | stale
formation_event_ref
formed_unit_versions
observation_refs
choice_event_ref | nil
runtime_reconciliation_ref
manifest_material_refs
basis_truth_statuses
event_truth_status = runtime_confirmed
```

The assessment event is runtime-confirmed. Plan content remains
`semantic_proposal` or its inherited content status.

☱ does not execute plan items, ask the substrate, choose another alternative,
or manifest output.

## Phase C: △ Plan Delivery

Only a current `complete` assessment creates terminal-boundary pressure from ☱
to △.

The `plan_delivery` action names the exact assessment and all material needed
for assembly. MANIFEST readiness must verify that the assessment still matches
the current Packet. Caller `options.result`, prior tick payloads and arbitrary
source refs are not readiness evidence.

MANIFEST assembles a deterministic Packet-local artifact:

```text
kind = manifest_payload
output.type = plan
output.status = complete
output.structure_protocol = plan.result.v0
output.shape = exact formation shape
output.items = current deliverable items in formation order
output.connections = bounded declared connections
output.selection = selected/suppressed partition when applicable
sources = exact formation/observation/choice/runtime refs
assembly.input_provenance = packet_state
assembly.rule = plan_delivery.v0
residue = visible loss and suppressed alternatives
terminal_cause = complete
truth_status = runtime_confirmed
content_truth_status = inherited proposal status
```

The body confirms the assembly event and terminal transition. It does not
promote the semantic truth of item values.

For compatibility, legacy MANIFEST assembly may remain callable without a
qualified action. It cannot count as exact plan-delivery evidence.

## Discharge And Re-Arm

```text
complete assessment written for exact state
  -> same plan_completion_review need disappears

current complete assessment reaches △ and Packet dies
  -> same plan_delivery need has no living successor

formed unit version changes before △
  -> assessment becomes stale, delivery disappears, upper/review pressure re-arms

new exact formation arrives
  -> old assessment cannot deliver the new form

choice partition changes
  -> old assessment becomes stale

work mode mismatch
  -> invariant failure at birth or action validation
```

No mutable `completion=true` flag is introduced.
