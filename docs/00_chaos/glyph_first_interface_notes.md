# Glyph-First Interface Notes

Raw note from the runtime crystallization pass.

## Pressure

Human text keeps needing grammar endings:

```text
look at the table and make the runtime crystal
```

Glyph pressure says the same shape more directly:

```text
⊞ -> ◈☱
```

The glyph form is not decoration.
It is a compressed process instruction.

## Working Reading

```text
⋯
  chaos pressure

☴
  observe/read without pretending stability

⊞
  table/yellowprint

◈
  crystall/blueprint

△
  manifest/output
```

Examples:

```text
⋯☴⊞☴◈☴
  run the documentation/build compression loop

⊞ -> ◈☱
  read table and produce a RUNTIME crystal

◈☱ -> △☱
  read RUNTIME crystal and think about its manifestation
```

## Interface Direction

For now, user and substrate may use words:

```text
flow
runtime
manifest
crystal
table
```

But body-native interfaces should become glyph-first over time.

Words can remain aliases.
Glyphs should become the primary routing surface.

## Why

Glyphs reduce semantic drift:

```text
runtime
  can be misread as memory, environment, execution, state, or backend

☱
  points to one operator slot in topology
```

Glyphs also avoid human grammar pressure.

In code, endings are not needed.
The body can route by operator and layer directly.

## Boundary

Glyph-first does not mean opaque.

Each glyph still needs:

```text
topology
contract
payload shape
test surface
```

Otherwise glyphs become magic words instead of machine routing.

## Open Shape

Future CLI could accept compact process commands:

```text
proc17 ⋯☴⊞☴◈☴
proc17 "⊞ -> ◈☱"
proc17 "◈☱ -> △☱"
```

This is not a current implementation target.
It is interface pressure.
