# Encode Code-Only Boundary

Status: design pressure.

This document records a major fork discovered from the cognitive battery.

## Decision

`☵ ENCODE` must not grow into a general human-language interpreter.

proc-17 is a body for coding work. It should encode code-shaped, runtime-shaped, and engineering-shaped material.

It should not try to extract deep truth from human philosophical prose by default.

## Why This Matters

The user koan battery showed that fluent prose can look meaningful while remaining non-operational for the body.

When a substrate returns one dense paragraph, current `☵` sees:

```text
semantic_line_field:
  item 1 = entire paragraph
```

Then `☳` has no real choice:

```text
selected = entire paragraph
killed = none
loss = 0
```

This is not useful collapse. It is a recorded trace, not body cognition.

One possible path would be to make `☵` parse prose into claims, metaphors, relations, examples, and residues.

We reject that as the default path.

That path would turn proc-17 into a philosophical-text interpreter. It would invite external vocabularies, metaphor parsing, therapeutic language, and aesthetic fluency. That is not the body we are building.

## Boundary

`☵` encodes inspectable or executable structure.

Valid default material:

- files
- paths
- diffs
- tests
- stdout/stderr
- errors
- JSON/Lua/table structures
- CLI args
- repo state
- runtime evidence
- constraints
- candidate actions
- missing organs
- unsupported forms
- engineering pressure

Invalid default material:

- dense philosophical prose
- poetic explanation
- therapeutic framing
- metaphysical interpretation
- generic cultural references
- beautiful but non-operational language

Invalid does not mean worthless.

It means: not body-field by default.

## Prose Handling

Human prose should enter the body conservatively.

Default:

```text
human prose -> semantic_proposal/raw_text
```

If it contains explicit engineering pressure:

```text
technical requirement -> extracted_candidate
constraint -> field item
runtime evidence -> field item
task instruction -> candidate action
```

Everything else remains raw semantic material.

`☵` should not pretend that a beautiful paragraph is an operational field.

## What This Preserves

This boundary narrows the solution space.

That narrowing is useful pressure.

It prevents these bad paths:

- building a generic philosophy parser
- turning `☵` into a second substrate
- making `☳` choose between poetic fragments
- treating human-language fluency as runtime structure
- letting koan answers become fake engineering truth

It preserves these good paths:

- code-first body
- runtime-confirmed field construction
- visible loss
- explicit uncertainty
- clean separation between procesis as soul and proc-17 as body

## Relation To Procesis

procesis may contain philosophical depth.

proc-17 should not re-derive that depth from arbitrary prose.

proc-17 should receive procesis as structured pressure, not as free-form text to interpret endlessly.

The body does not need to understand all human language.

The body needs to know what can become code, test, trace, constraint, action, or residue.

## Working Rule

Before `☵` encodes something as field, ask:

```text
Can this item be inspected, executed, tested, constrained, selected, rejected, or manifested?
```

If yes, it may enter the field.

If no, keep it as raw semantic proposal.

## Consequence

The user koan battery is still useful.

But its role changes.

It is not a target for philosophical understanding.

It is a boundary test:

```text
Can proc-17 avoid turning fluent human prose into fake operational structure?
```

That is the real test.
