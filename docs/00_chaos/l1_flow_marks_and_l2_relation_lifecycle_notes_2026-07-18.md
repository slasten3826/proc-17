## 3. L1 As Continuing Flow, Not Packet Identity

The first fluid FLOW implementation in `slop.raw.txt` separated three ideas,
although it did not yet turn them into a clean contract:

```text
process essence
hash of one flow moment
process signature accepted by emergent relating
```

Its decisive comment was:

```text
the hash is not identity; it is a moment of flow
```

The new hypothesis is therefore not:

```text
L1 computes hash(prompt)
L1 allocates a UUID
L1 samples an unrelated random number
```

It is:

```text
L1 continues to move
a Packet is born at a particular state of that movement
FLOW records a bounded mark of that birth moment
the Packet carries the mark but does not become the L1 process
```

### 3.1 Three-body analogy

The useful analogy is deterministic chaos, such as the mathematical
three-body problem:

```text
the evolution has a law
the same exact state and schedule reproduce the same trajectory
a small perturbation may produce a rapidly diverging trajectory
the observed result may look random without being an external RNG draw
```

Implementation still requires an exact algorithm. The important distinction
is that the mark is not assigned by a semantic naming algorithm. It is measured
from the current physical trajectory.

Therefore the mark is better called provisionally:

```text
flow_mark
```

rather than:

```text
packet_identity
cryptographic_signature
content_hash
```

### 3.2 Uniqueness is an event property

The standalone L1 fingerprint is bounded by `MOD = 59049`. Numeric collisions
are lawful and unavoidable. The fingerprint alone cannot guarantee global
uniqueness, authenticity, or security.

If exact birth distinction is required, it must come from the event envelope:

```text
L1 stream/epoch reference
flow tick or birth sequence
bounded L1 snapshot/fingerprint
source adapter provenance
protocol version
```

The event can be unique even when one bounded measurement repeats. Adding an
administrative sequence does not make the physical fingerprint less honest;
it only prevents audit records from confusing two moments.

## 4. Lifecycle Conflict Exposed By The Eternal L1 Hypothesis

The current L1 boundary table provisionally says:

```text
L1 advances at most once per committed Packet body tick
L1 freezes on Packet terminal state
```

The continuing-flow hypothesis says something different:

```text
L1 outlives an individual Packet
the Packet receives a bounded birth mark or local projection
Packet death freezes the carried record, not necessarily the source process
```

These laws cannot be silently treated as equivalent.

Possible future resolution:

```text
continuing L1 process     survives Packet lives
Packet-local flow mark   immutable after birth
optional local L1 view   bounded episode tied to one Packet and freezable
```

This is not yet selected. The standalone `l1.freeze()` contract remains valid
for the standalone state and its tests. Its future production ownership is now
OPEN and requires a table amendment before integration.

## 5. The Signature Does Not Police CONNECT

The flow mark is not a whitelist saying which Packet may connect to which
other Packet.

L2 CONNECT does not connect parent Packets to children and does not establish
lineage. Parent/corpse/carrier continuity belongs to lineage and NETWORK
boundaries.

Within L2, CONNECT works on distinguished material inside the current form
domain. The flow mark may later supply provenance or initial conditions, but
it does not itself grant relation authority.

This corrects the rejected hypothesis:

```text
rejected:
  equal or compatible L1 signatures authorize Packet-to-Packet CONNECT
```

## 6. CONNECT Is A Transient Relation Sensor

The older packet model states the contract directly:

```text
CONNECT = resonance snapshot
detects structure, does not preserve it
state = none
write = E_edges_raw
```

This means `☰` does not connect in the durable sense. It detects or briefly
holds a possible relation and offers that relation to the next operator.

The Packet Tree gives CONNECT four neighbors:

```text
▽ FLOW
☷ DISSOLVE
☴ OBSERVE
☵ ENCODE
```

The edge is not only permission to move. It describes the possible fate of a
transient relation.

## 7. Four Roads Of CONNECT

### 7.1 FLOW and CONNECT: relation emergence or return

```text
▽ -> ☰
```

Raw flow begins to exhibit relation. CONNECT distinguishes a resonance without
yet preserving it.

The reverse execution direction remains to be tabled precisely:

```text
☰ -> ▽
```

The current hypothesis is that an unretained relation returns to continuing
flow or re-enters raw emergence. It must not become durable merely because
CONNECT observed it.

### 7.2 CONNECT to DISSOLVE: relation release

```text
☰ -> ☷
```

The transient relation is weakened, separated, or returned as residue. It is
not first required to become a permanent active relation merely so DISSOLVE
can release it.

The exact distinction between:

```text
discarding a raw snapshot
weakening a formed relation
irreversibly dissolving encoded form
```

remains a table question. These acts may have different loss laws even though
they use the same glyph at different layer phases.

### 7.3 CONNECT to OBSERVE: relation becomes visible

```text
☰ -> ☴
```

OBSERVE receives a bounded view of the transient relation and records what was
seen. Observation does not activate, stabilize, or own the relation.

This road can make relation evidence auditable without making it persistent.

### 7.4 CONNECT to ENCODE: relation becomes form

```text
☰ -> ☵
```

This is the only CONNECT road that may "pour concrete" around a relation.

ENCODE consumes selected raw relation structure and produces a new retained
form under explicit remapping and loss. The raw relation need not survive with
the same identity. What persists is the encoded form and its provenance.

The later packet-slop L2 branch independently found the same dependency as:

```text
hidden CONNECT inside ENCODE
```

In that model ENCODE is a process, not a button. CONNECT continuously holds the
channel between CHAOS and the forming CALM state. If the channel is lost, there
is nothing left to encode.

## 8. Reconciled Ownership Hypothesis

A first ownership split now looks like:

| Surface | Meaning | Candidate owner |
|---|---|---|
| raw relation snapshot | transient resonance in the current field epoch | `☰ CONNECT` |
| observed relation evidence | bounded audit projection, no persistence right | `☴ OBSERVE` |
| encoded structure/remap | retained form produced with explicit loss | `☵ ENCODE` |
| relation release/residue | weakening or dissolution under a typed reason | `☷ DISSOLVE` |
| momentum/habit | persistence or reinforcement of already formed structure | `☱ RUNTIME` |

The critical distinction is:

```text
ENCODE originates retained form from transient relation.
RUNTIME may preserve or reinforce form; it must not invent the relation or
launder a raw snapshot into form merely by observing time.
```

This reconciles the new L2 reading with the older UPM statement that RUNTIME
owns momentum. Cement and inertia are different operations.

The CONNECT storage contract is tested, but the relation lifecycle is not yet

## 10. Why Earlier CONNECT Witness Work Stalled

The project repeatedly tried to answer:

```text
which consumer requests CONNECT?
```

That question assumed CONNECT was a durable graph writer needing one later
consumer.

The recovered model asks a more exact question:

```text
which neighboring operator receives this transient relation, and what right
does that road have over it?
```

The four neighbors are the named readers. They do not all read the same thing
for the same purpose:

```text
FLOW      continuation/re-emergence
DISSOLVE  release
OBSERVE   visibility
ENCODE    retention as form
```

The previous relation-consumer work is not discarded. Its versioned coverage,
same-ref, readiness, and unsupported-result laws remain useful for discrete
L2 operations. What changes is the assumption that every useful CONNECT must
eventually create an active durable edge.

## 11. Falsifiers For The New Reading

The hypothesis must be killed if implementation requires any of these lies:

```text
raw CONNECT snapshot persists forever without a receiving operator
OBSERVE makes a relation active merely by recording it
DISSOLVE must first activate a raw relation before it can reject/release it
RUNTIME invents a relation that ENCODE never formed
ENCODE ignores the relation snapshot but still claims relation-guided form
L1 fingerprint is called globally unique despite lawful MOD collisions
flow mark is called cryptographic authenticity
Packet lineage is inferred from CONNECT resonance
```

Positive matched cases for a future table:

```text
C1  CONNECT only                   raw epoch exists; no retained form
C2  CONNECT -> OBSERVE             evidence exists; no retained form
C3  CONNECT -> DISSOLVE            raw candidate is released/denied honestly
C4  CONNECT -> ENCODE              retained form exists with provenance + loss
C5  C4 -> RUNTIME                  formed relation gains momentum, not origin
C6  remove raw relation from C4    ENCODE cannot claim relation-guided form
```

For L1 flow marks:

```text
F1  same exact L1 state/schedule   same trajectory and mark projection
F2  one bounded perturbation       deterministic divergence becomes visible
F3  repeated bounded fingerprint   birth events remain distinguishable
F4  remove flow mark               no hidden routing or semantic claim changes
```

F4 is important: the first flow-mark implementation may establish provenance
without yet gaining routing authority.
